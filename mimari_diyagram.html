CloudFront CDN
AWS · Edge / CDN Katmanı
Ne yapar?

Tüm trafiğin ilk giriş noktası. Türkiye'den bağlanan kullanıcıları İrlanda sunucusuna göndermek yerine Frankfurt edge'ine yönlendirir. ~120ms yerine ~18ms.

Path routing

/api/* → API Gateway → Lambda
/* → S3 React build

Ek görevler

WAF (bot engelleme, rate limit), wildcard SSL (ACM), ürün görseli cache (7 gün TTL), satıcı sitelerini host etme.

Maliyet
~$1/ay

1TB/ay free data transfer. Türkiye trafiği Frankfurt'tan çıkar.

Bağlantılar
S3 React API Gateway ACM S3 Assets
API Gateway
AWS · REST API · Giriş Noktası
Ne yapar?

CloudFront'tan gelen /api/* isteklerini ilgili Lambda fonksiyonuna yönlendirir. Her endpoint için ayrı Lambda tetiklenir.

Özellikler

Request validation, JWT authorizer, rate limiting, CORS headers, request/response mapping.

Maliyet
~$0/ay

1M istek/ay free tier. Aşarsa $3.50/1M istek.

RDS PostgreSQL
db.t3.micro · Single-AZ · + pgvector
Ne tutar?

Kullanıcılar, satıcılar, ürünler, siparişler, para işlemleri (transactions), site konfigürasyonları.

pgvector eklentisi

Ürün açıklamalarının embedding vektörleri bu veritabanında saklanır. RAG araması için cosine similarity sorgusu çalıştırılır. Ayrı bir vektör DB gerekmez.

Büyük veri stratejisi

orders ve transactions tabloları aylık partition'a bölünmeli. products tablosunda seller_id + category composite index şart.

Maliyet
~$13/ay ⚠️

En büyük maliyet kalemi. Multi-AZ $28/ay olur — MVP için Single-AZ yeterli.

pgvector
PostgreSQL eklentisi · RAG mimarisi
Neden gerekli?

Chatbot "kırmızı spor ayakkabı" dediğinde 1 milyon ürünü LLM'e veremezsin. pgvector ile önce en yakın 5 ürün bulunur, sadece onlar Claude'a gönderilir.

Akış

1. Ürün eklenince → embedding üret (Claude Embeddings API)
2. Vektörü RDS'e kaydet
3. Kullanıcı sorusu → vektöre çevir → cosine similarity → top-5
4. top-5 ürün + soru → Claude → cevap

Maliyet
RDS içinde ~$0 ek

Sadece embedding üretme (Claude API token maliyeti).

DynamoDB
AWS · NoSQL · Session & Chat
Ne tutar?

Kullanıcı session'ları, sepet verileri, chatbot konuşma geçmişleri, öneri cache sonuçları, rate limit sayaçları.

Neden RDS değil?

Chat geçmişi JSON blob — relational DB'de tutmak hem pahalı hem yavaş. DynamoDB'de her kullanıcının konuşması ayrı item, TTL ile otomatik silinir.

Maliyet
~$0/ay

25GB + 25 WCU + 25 RCU free tier. Büyük ihtimalle bu sınırı aşmazsın.

SQS — Finance Queue
Finansal işlemler için kritik kuyruk
Akış

EasyPost → Webhooks λ → SQS Finance → Finance Worker λ → Stripe API

Neden şart?

Stripe o an yavaşsa veya çökse bile EasyPost'a anında 200 OK dönersin. Mesaj SQS'te bekler. Stripe düzelince otomatik işlenir. Para kaybolmaz.

Güvenlik mekanizmaları

DLQ: 3 başarısız denemeden sonra Dead Letter Queue'ya düşer, alarm çalar.
Idempotency key: order_id + event_type kombinasyonu — aynı transfer iki kez yapılmaz.
Visibility timeout: İşlenirken başka worker alamaz.

Maliyet
~$0/ay

1M mesaj/ay free tier.

SQS — Publish Queue
Site yayınlama işleri
Neden gerekli?

Site render işlemi 5-30 saniye sürer. Kullanıcıyı bu kadar bekletirsen ürün kullanılmaz. SQS ile kullanıcı anında geri döner, render arka planda tamamlanır.

Akış

Kullanıcı "Yayınla" → Orders λ → SQS Publish → "Yayınlanıyor..." göster → Publisher λ arka planda render → S3'e yaz → "Siteniz yayında!" bildir

Vision λ
Görme engelli kullanıcı desteği
Ne yapar?

Ürün sayfasına girildiğinde görsel + ürün adı + fiyat bilgisini alır, Claude Vision API'ye gönderir. Dönen metni AWS Polly ile Türkçe sese çevirir.

Örnek çıktı

"Bu Ülker marka bir çikolata. Görselde 6 adet görünse de bir adet ürün 12 TL. Satın almak veya sepete eklemek ister misiniz?"

Cache stratejisi

Aynı image_hash için DynamoDB'de cache tutulur. Popüler ürünler için API maliyeti sıfıra yaklaşır.

Bağlantılar
Claude Vision AWS Polly DynamoDB cache
Sipariş Chatbot λ
Kullanıcı sipariş sorgulama botu
Ne yapar?

Kullanıcı "siparişim nerede?" dediğinde EasyPost'tan kargo durumunu çeker, Claude API ile doğal dil cevabı üretir.

Konuşma geçmişi

Her turn DynamoDB'ye yazılır. TTL ile 30 gün sonra silinir. Kullanıcı sayfayı kapayıp gelince konuşmaya devam edebilir.

SSE stream

Claude'un cevabı token token frontend'e akar. Kullanıcı bekleme hissetmez.

Ürün Q&A λ
RAG tabanlı ürün chatbotu
RAG akışı

1. Kullanıcı sorusu → embedding
2. pgvector cosine search → top-5 ürün
3. top-5 + soru → Claude API
4. Cevap stream → frontend

Neden RAG?

1M ürünü LLM'e veremezsin (context window + maliyet). pgvector ile sadece ilgili 5 ürünü seçip gönderirsin — doğru ve ucuz.

Site Builder Agent λ
AI ile site tasarımı
Akış

1. Kullanıcı prompt girer → API Gateway → SQS async
2. Agent λ tetiklenir → Claude API (multi-turn)
3. JSON layout üretilir → HTML/CSS render
4. S3'e yazılır → CF invalidate
5. SSE ile "Header tamamlandı... Ürün grid ekleniyor..." bildirimleri

Neden async?

Bu işlem 30-120 saniye sürebilir. Lambda timeout 15 dakika — yeterli. Ama kullanıcıyı beklettirmemek için SQS + SSE polling şart.

Öneri λ
Collaborative filtering sistemi
Nasıl çalışır?

EventBridge'den kullanıcı tıklama ve sipariş eventlerini alır. "Bu ürünü alanlar şunu da aldı" matrisini DynamoDB'de günceller. Sonuçları TTL ile cache'ler.

Amazon Personalize yerine neden bu?

Personalize kurulumu karmaşık, aylık maliyeti yüksek, minimum veri seti gerektirir. Bu yaklaşım $0 maliyetle başlar, yeterli veri birikince Personalize'a geçilebilir.

EasyPost webhook akışı

1. POST isteği gelir
2. Signature doğrula (HMAC)

Neden bu kadar hızlı dönmeli?

EasyPost 5 saniye timeout uygular. Cevap gelmezse aynı webhook'u 10 kez gönderir. Senkron Stripe çağrısı yapsan timeout kaçınılmaz.

Finance Worker λ
Escrow → Satıcı transfer işlemi
Akış

1. SQS Finance'ten mesaj al
2. DynamoDB'de idempotency key kontrol et
3. Daha önce işlendiyse atla
4. Stripe Transfer API çağır
5. Başarılı → mesajı SQS'ten sil + DynamoDB'ye işle
6. Başarısız → mesajı silme, SQS tekrar dener

Kritik: idempotency

order_id + "delivered" kombinasyonu DynamoDB'de işaretlenir. EasyPost aynı eventi 2 kez gönderse bile transfer bir kez yapılır.
